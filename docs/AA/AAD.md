# AAD — Консолидированное описание архитектуры

## Описание

Документ объединяет все архитектурные решения ([ADL.md](ADL.md)) в связное описание архитектуры системы AI Assistants Proxy. Каждое решение вносит вклад в определённый аспект архитектуры; вместе они формируют целостный архитектурный образ системы.

## Архитектурная концепция

AI Assistants Proxy — корпоративный сервис проксирования доступа к западным AI API ([C1-BC-001](C1/C1-BC-001.md)). Архитектура системы строится на следующих ключевых решениях:

### 1. Двухуровневая архитектура ([ADR-000001](ADR/ADR-000001.md))

Фундаментальное архитектурное решение — разделение на два физически разнесённых уровня:

- **Access tier (РФ):** точка входа для пользователей. Аутентификация, авторизация, журналирование, кеширование, VRRP failover. 2 ноды. Squid за nginx SNI Router (:443 → 127.0.0.1:3128).
- **Upstream tier (вне РФ):** анонимный транспорт. Минимальная конфигурация, без логирования, без аутентификации. 1 нода. Squid на порту 80 напрямую (без nginx).

Разделение обусловлено необходимостью обхода гео-ограничений AI-сервисов при сохранении контроля доступа и анонимности на территории РФ.

### 2. Squid как прокси-движок ([ADR-000002](ADR/ADR-000002.md))

Оба уровня построены на базе **Squid** — зрелого HTTP/HTTPS прокси-сервера с поддержкой:
- CONNECT-туннелирования (TLS passthrough без расшифровки)
- Basic Auth через ncsa_auth
- ACL-фильтрации доменов (dstdomain, dstdom_regex)
- Кеширования (ufs)
- SMP-параллелизма (workers)
- Каскадирования (cache_peer)

Access-прокси каскадирует запросы на upstream через `cache_peer` с балансировкой `userhash`.

### 3. Отказоустойчивость ([ADR-000006](ADR/ADR-000006.md))

На access-уровне реализован VRRP failover через **Keepalived**:
- Виртуальный IP (VIP) — единая точка входа
- Автоматическое переключение MASTER → BACKUP при отказе
- Health-check: мониторинг процесса Squid (vrrp_script)

На upstream-уровне балансировка реализована через Squid `userhash`:
- Распределение запросов между upstream-нодами
- Автоматический failover при недоступности peer

### 4. Безопасность ([ADR-000004](ADR/ADR-000004.md))

Многослойная защита:
- **CrowdSec** — IPS с коллаборативной репутационной базой (все ноды, whitelist инфраструктуры)
- **UFW + nftables** — межсетевое экранирование (UFW на upstream, CrowdSec nftables bouncer на всех)
- **Squid ACL** — фильтрация по белому списку доменов
- **Анонимизация** — удаление заголовков X-Forwarded-For, Via на upstream

### 5. Co-deployment ([ADR-000003](ADR/ADR-000003.md), [ADR-000005](ADR/ADR-000005.md))

Upstream-ноды используются совместно с другими проектами:
- **nginx SNI Router** — L4-маршрутизация TLS-потоков по SNI на порту 443
- **MTProxy (mtg)** — прокси для Telegram с fake-TLS маскировкой

На upstream nginx:443 маршрутизирует только MTProxy (Squid на порту 80 напрямую — ssl_preread несовместим с plain HTTP). На access nginx:443 маршрутизирует и Squid, и MTProxy.

### 6. Infrastructure as Code ([ADR-000007](ADR/ADR-000007.md))

Вся инфраструктура управляется через **Ansible**:
- Единый `playbook.yml` — точка входа
- Роли: `dns_resolver`, `crowdsec`, `nginx`, `mtproxy`; inline-задачи для Squid, Keepalived, клиентских конфигураций
- Идемпотентное выполнение
- Inventory-driven конфигурация
