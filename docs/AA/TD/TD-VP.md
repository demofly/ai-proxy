# TD-VP — Viewpoints (представления стейкхолдеров)

## Описание

Архитектурные представления (viewpoints) определяют, какие аспекты системы видит каждый стейкхолдер из [C1-BC-002](../C1/C1-BC-002.md). Каждый viewpoint описывает набор артефактов, релевантных для данного стейкхолдера, и ключевые архитектурные решения, затрагивающие его интересы.

## Матрица стейкхолдеров и viewpoints

| ID | Стейкхолдер | Viewpoint | Что видит | Ключевые артефакты |
|---|---|---|---|---|
| SH-01 | Владелец продукта | Стратегический | Доступность сервиса, затраты на инфраструктуру, KPI, соответствие бизнес-целям | [C1-BR.md](../C1-BR.md), [C1-BC-004](../C1/C1-BC-004.md), [Ideas.md](../Ideas.md) |
| SH-02 | DevOps-инженер | Операционный | Конфигурация компонентов, развёртывание, мониторинг, troubleshooting | [C3-CR.md](../C3-CR.md), [C4-SR.md](../C4-SR.md), playbook.yml, inventory.yml |
| SH-03 | Инженер-разработчик | Потребительский | Настройка прокси в IDE/браузере, стабильность соединения, эндпоинт | [C2-FR-007](../C2/C2-FR-007.md), Clients/ |
| SH-04 | Служба ИБ | Безопасность | Аудит доступа, IPS-события, анонимизация заголовков, firewall-правила | [C2-NF-002](../C2/C2-NF-002.md), [CC-Risks.md](../CC/CC-Risks.md) |

## Детальные viewpoints

### VP-01: Стратегический (Владелец продукта — SH-01)

**Фокус:** достижение бизнес-целей, контроль затрат, оценка рисков.

**Ключевые вопросы:**
- Обеспечена ли доступность AI-инструментов для инженерной команды?
- Каковы затраты на инфраструктуру (4 сервера)?
- Какие риски существуют и как они митигируются?
- Каков roadmap развития системы?

**Артефакты:**

| Артефакт | Что даёт стейкхолдеру |
|---|---|
| [C1-BC-004](../C1/C1-BC-004.md) — Бизнес-цели и KPI | Целевые метрики и текущее состояние |
| [C1-BR.md](../C1-BR.md) — Реестр бизнес-контекста | Полная картина системы |
| [Ideas.md](../Ideas.md) — Gaps и идеи | Нереализованные возможности и пробелы |
| [CC-Risks.md](../CC/CC-Risks.md) — Риски | Реестр рисков с митигациями |
| [ADL.md](../ADL.md) — Архитектурные решения | Обоснование принятых решений |

**Связанные ADR:** [ADR-000001](../ADR/ADR-000001.md) (архитектура), [ADR-000007](../ADR/ADR-000007.md) (IaC).

---

### VP-02: Операционный (DevOps-инженер — SH-02)

**Фокус:** развёртывание, конфигурирование, мониторинг, обновление.

**Ключевые вопросы:**
- Как развернуть / обновить систему?
- Как добавить нового пользователя?
- Как масштабировать (добавить ноды)?
- Как диагностировать проблемы?
- Какие компоненты и роли Ansible используются?

**Артефакты:**

| Артефакт | Что даёт стейкхолдеру |
|---|---|
| playbook.yml, inventory.yml | Точки входа для развёртывания |
| Ansible roles (roles/) | Конфигурация каждого компонента |
| [TD-FA.md](TD-FA.md) — Функциональные области | Группировка компонентов по назначению |
| [TD-TA.md](TD-TA.md) — Technology Architecture | Каталог технологий и версий |
| Каталог наблюдаемости (TD-FA.md §Events, §Metrics) | Что мониторить и где искать |

**Связанные ADR:** [ADR-000002](../ADR/ADR-000002.md) (Squid), [ADR-000006](../ADR/ADR-000006.md) (Keepalived), [ADR-000007](../ADR/ADR-000007.md) (Ansible).

**Операционные сценарии:**

| Сценарий | Действие | Команда / Инструмент |
|---|---|---|
| Развёртывание с нуля | Запуск полного playbook | `ansible-playbook -i inventory.yml playbook.yml` |
| Добавление пользователя | Обновление inventory → целевой тег | `ansible-playbook -i inventory.yml playbook.yml --tags squid_access` |
| Проверка состояния | Статус сервисов | `systemctl status squid`, `cscli metrics`, `squidclient mgr:info` |
| VRRP failover | Автоматический; ручной — остановка Keepalived | `systemctl stop keepalived` на MASTER |
| Обновление CrowdSec | Обновление коллекций | `cscli hub update && cscli collections upgrade --all` |

### VP-03: Потребительский (Инженер-разработчик — SH-03)

**Фокус:** настройка прокси, стабильность работы AI-инструментов.

**Ключевые вопросы:**
- Какой адрес и порт прокси?
- Как настроить IDE / браузер / CLI?
- Какие AI-сервисы доступны через прокси?
- Что делать, если прокси не работает?

**Артефакты:**

| Артефакт | Что даёт стейкхолдеру |
|---|---|
| [C2-FR-007](../C2/C2-FR-007.md) — Конфигурации клиентов | Инструкции по настройке |
| Clients/ (в репозитории) | Готовые конфигурации для VSCode, FoxyProxy, PowerShell |
| [C1-BC-003](../C1/C1-BC-003.md) — Внешние системы | Список доступных AI-сервисов |

**Точка подключения:**
- Адрес: виртуальный IP access-прокси (VRRP VIP)
- Порт: 3128 (Squid HTTP proxy port)
- Аутентификация: Basic Auth (логин/пароль от DevOps-инженера)

### VP-04: Безопасность (Служба ИБ — SH-04)

**Фокус:** аудит, контроль доступа, обнаружение угроз, анонимизация.

**Ключевые вопросы:**
- Кто и когда обращался к AI API?
- Как контролируется утечка данных?
- Какие механизмы IPS используются?
- Как анонимизируются исходящие заголовки?
- Какие firewall-правила действуют?

**Артефакты:**

| Артефакт | Что даёт стейкхолдеру |
|---|---|
| [C2-NF-002](../C2/C2-NF-002.md) — Безопасность | Требования к безопасности |
| [C2-FR-005](../C2/C2-FR-005.md) — Журналирование | Формат и хранение аудит-логов |
| [CC-Risks.md](../CC/CC-Risks.md) — Риски | Реестр рисков безопасности |
| [ADR-000004](../ADR/ADR-000004.md) — CrowdSec | Обоснование выбора IPS |

**Механизмы безопасности:**

| Механизм | Компонент | Описание |
|---|---|---|
| Аутентификация (Basic Auth) | Squid Access | ncsa_auth helper, файл passwd |
| Фильтрация доменов (whitelist) | Squid Access | ACL dstdomain + url_regex |
| IPS / обнаружение вторжений | CrowdSec + nftables | Сценарии обнаружения → решения → блокировка IP |
| Анонимизация заголовков | Squid Upstream | Удаление X-Forwarded-For, Via; forwarded_for off |
| Межсетевое экранирование | nftables | Ограничение входящих соединений на upstream-нодах |
