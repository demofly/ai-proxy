## Описание

Squid Access Proxy -- основной компонент access tier, развёрнутый на серверах в Российской Федерации. Обеспечивает точку входа для пользователей и реализует следующие функции:

1. **Аутентификация** -- Basic Auth через `basic_ncsa_auth` с файлом паролей `/etc/squid/passwd`. Параметры: 5 дочерних процессов (startup=5, idle=1), TTL учётных данных 2 часа, регистрозависимый ввод.

2. **Фильтрация доменов** -- два механизма ACL:
   - `dstdomain` -- явный список разрешённых доменов (inventory: `allowed_domains`)
   - `dstdom_regex` -- регулярные выражения для доменных зон (inventory: `allowed_domain_patterns`, например `\.ai$` для всех доменов зоны `.ai`)

3. **Кеширование** -- дисковый кеш `ufs` в `/var/spool/squid` (100 MB, 16 L1-директорий, 256 L2-директорий). Максимальный размер объекта 50 MB. Негативный TTL = 0 (ответы 4xx/5xx не кешируются). Кеш в памяти -- 60% от лимита памяти (20% RAM сервера), LRU-политика, максимум 512 KB на объект в RAM.

4. **Журналирование** -- `access_log` в формате squid (daemon mode), `cache_log` для диагностики. Буферизация логов включена (`buffered_logs on`) для снижения I/O.

5. **Цепочка к upstream** -- `cache_peer` с балансировкой `userhash` (sticky sessions по имени пользователя, fallback на `sourcehash`). Параметры отказоустойчивости: `connect-fail-limit=2`, `connect-timeout=5`, `dead_peer_timeout=15 seconds`. Все запросы маршрутизируются через upstream (`never_direct allow all`).

6. **Анонимизация** -- подмена User-Agent на Chrome 138, добавление заголовков браузера (Accept, Accept-Language, Sec-*), удаление прокси-заголовков (Via, X-Forwarded-For, X-Real-IP, X-Originating-IP).

7. **Производительность** -- SMP workers = количество ядер CPU, динамический расчёт памяти (20% RAM), persistent connections, pipelining (prefetch=10), retry on error (max 3 попытки).

## Механизм реализации

### Механизм обработки клиентского запроса

Обработка каждого клиентского запроса проходит следующую последовательность:

1. Клиент отправляет HTTP CONNECT на порт 443. nginx SNI Router передаёт соединение на Squid (127.0.0.1:3128) с PROXY protocol.
2. Squid проверяет аутентификацию (Basic Auth) через `basic_ncsa_auth`.
3. Squid проверяет целевой домен против ACL `allowed_domains` и `allowed_domain_regexp`.
4. При совпадении -- запрос маршрутизируется к upstream cache_peer по алгоритму `userhash`.
5. Upstream-прокси устанавливает соединение с целевым AI API.
6. TLS-туннель устанавливается end-to-end (access-прокси не видит содержимое).
7. Факт обращения фиксируется в access.log (при включённом журналировании).

### Механизм отказоустойчивости upstream-цепочки

1. При сбое соединения с upstream -- Squid увеличивает счётчик `connect-fail-limit`.
2. После 2 неудачных попыток (`connect-fail-limit=2`) upstream помечается как dead.
3. Трафик перенаправляется на оставшиеся upstream-прокси.
4. Каждые 15 секунд (`dead_peer_timeout`) Squid проверяет доступность dead peer.
5. При восстановлении -- upstream возвращается в пул.

### Механизм динамического масштабирования ресурсов

1. Ansible определяет количество ядер CPU (`ansible_processor_vcpus`) и объём RAM (`ansible_memtotal_mb`).
2. SMP workers = CPU cores (параллельная обработка запросов).
3. `memory_pools_limit` = 20% RAM (общий лимит памяти Squid).
4. `cache_mem` = 60% от лимита (кеш объектов в RAM).
5. Файловые дескрипторы = max(4096 * CPU, 65535).
6. Сетевые буферы, очереди и TCP-параметры масштабируются пропорционально CPU и RAM.

### Контрактное

- **Входной контракт:** HTTP CONNECT запрос с Basic Auth (login:password в base64) через nginx SNI Router (:443 → 127.0.0.1:3128, PROXY protocol). Proxy-Authorization header обязателен.
- **Выходной контракт:** TLS tunnel к целевому AI API через upstream cache_peer. Заголовки прокси удалены, User-Agent заменён.
- **Контракт с upstream:** cache_peer parent на порт 80 (plain HTTP CONNECT), балансировка userhash.
- **Контракт с CrowdSec:** access.log в формате squid (timestamp, response_time, client_ip, result_code, size, method, URL, user, hierarchy, content_type).

### Аналитическое

Выбор Squid обусловлен следующими факторами:
- Зрелый HTTP-прокси с нативной поддержкой CONNECT-туннелирования (25+ лет развития).
- Встроенная поддержка цепочки прокси (cache_peer) с балансировкой и failover.
- Гибкая система ACL (dstdomain, dstdom_regex, proxy_auth) без необходимости внешних модулей.
- SMP-режим (workers) для утилизации многоядерных серверов.
- Минимальный footprint: конфигурация через единственный текстовый файл, управление через systemd.

### Эмпирическое

Верификация выполняется Ansible-задачами:
- Валидация конфигурации: `squid -k parse -f /etc/squid/squid.conf` (при deploy).
- Функциональная проверка: curl через прокси с credentials к разрешённому домену.
- Проверка фильтрации: curl к запрещённому домену -> HTTP 403.
- Проверка отказоустойчивости: отключение upstream -> переключение на оставшийся.

| Риск | Митигация |
