## Описание

Ansible Deployment System -- IaC-компонент (Infrastructure as Code), обеспечивающий полностью автоматизированное развёртывание и конфигурирование всех серверных компонентов системы. Единая точка входа: `ansible-playbook -i inventory.yml playbook.yml`.

### Структура

```
Servers/deploy/
  playbook.yml          # Единый playbook для всех нод
  inventory.yml         # Inventory с группами access_proxies и upstreams
  passwd                # Файл паролей для Basic Auth
  debug-vars.yml        # Отладочные переменные
  templates/            # Jinja2-шаблоны (Squid, Keepalived, nftables)
    squid-access.conf.j2
    squid-upstream.conf.j2
    keepalived.conf.j2
    nftables-upstream.conf.j2
  roles/                # Ansible roles для модульных компонентов
    crowdsec/           # CrowdSec IPS
    nginx/              # nginx SNI Router
    mtproxy/            # MTProxy (mtg)
Clients/                # Клиентские конфигурации
  Claude/settings.json
  Gemini/settings.json
  FoxyProxy/FoxyProxy.json
  PowerShell/register-vars.ps1
  PowerShell/unregister-vars.ps1
```

### Inventory-группы

| Группа | Описание | Роль (`proxy_role`) |
|---|---|---|
| `access_proxies` | Access-прокси в РФ | `access` |
| `upstreams` | Upstream-прокси вне РФ | `upstream` |

### Playbook: фазы развёртывания

Playbook `playbook.yml` -- единый play для `hosts: all`, выполняющий развёртывание последовательно по фазам:

1. **System preparation** -- удаление ненужных пакетов (packagekit, multipath-tools), отключение UseDNS в sshd, установка базовых утилит (mc, haveged).

2. **Resource calculation** -- динамическое вычисление ресурсных лимитов на основе CPU и RAM: SMP workers, memory pools, FD limits, socket buffers, TCP queues, TIME_WAIT buckets.

3. **System tuning** -- sysctl-параметры для low-latency networking: TCP fast open, buffer scaling, SYN flood protection, FD limits через systemd override.

4. **Squid deployment** -- установка Squid, обнаружение nginx (co-deployment), развёртывание конфигурации (access или upstream), интеллектуальный reload/restart (на основе diff критических параметров), инициализация кеша, настройка firewall (upstream).

5. **Keepalived** -- установка и настройка VRRP failover (access-прокси, опционально).

6. **CrowdSec** -- `include_role: crowdsec` (при `crowdsec_enabled=true`).

7. **nginx** -- `include_role: nginx` (access-прокси при `nginx_on_443=true`).

8. **MTProxy** -- `include_role: mtproxy` (upstream-прокси при `mtproxy_enabled=true`).

### Клиентские конфигурации

Директория `Clients/` содержит готовые конфигурационные файлы для настройки клиентских приложений:

| Клиент | Файл | Описание |
|---|---|---|
| Claude (VSCode) | `Claude/settings.json` | Настройка HTTP proxy для Claude extension |
| Gemini | `Gemini/settings.json` | Настройка HTTP proxy для Gemini |
| FoxyProxy (браузер) | `FoxyProxy/FoxyProxy.json` | Профиль прокси для Firefox/Chrome |
| PowerShell | `PowerShell/register-vars.ps1` | Регистрация переменных окружения HTTP_PROXY/HTTPS_PROXY |
| PowerShell | `PowerShell/unregister-vars.ps1` | Удаление переменных окружения |

### Интеллектуальный reload/restart Squid

Playbook определяет, требуется ли restart (изменение `http_port` или `cache_dir`) или достаточно reload (все остальные изменения):

1. Сохраняются критические параметры текущего конфига (`squid_params_before`).
2. Развёртывается новая конфигурация.
3. Сравниваются новые параметры с сохранёнными (`squid_params_after`).
4. При расхождении -- restart; при совпадении -- reload (без обрыва соединений).

## Механизм реализации

### Механизм идемпотентного развёртывания

1. Ansible выполняет задачи последовательно, каждая задача идемпотентна (повторный запуск не меняет уже корректное состояние).
2. Шаблоны развёртываются с проверкой изменений (`register: result`, `when: result.changed`).
3. Squid config валидируется перед применением (`validate: 'squid -k parse -f %s'`).
4. Handlers вызываются только при фактических изменениях (`notify`).
5. Условная активация компонентов через переменные (`crowdsec_enabled`, `mtproxy_enabled`, `enable_keepalived`).

### Механизм параллельного развёртывания

1. Ansible подключается ко всем нодам параллельно (по умолчанию forks=5).
2. Playbook `hosts: all` -- все ноды получают базовую конфигурацию.
3. Задачи с `when: proxy_role == 'access'` выполняются только на access-нодах.
4. Задачи с `when: proxy_role == 'upstream'` выполняются только на upstream-нодах.
5. Roles включаются условно (when clause на include_role).

### Механизм автоопределения окружения

1. **nginx co-deployment** -- playbook проверяет наличие активного nginx с виртуалхостом на 443 (`systemctl is-active nginx + grep listen 443`). Результат: `nginx_on_443` (auto-detect или manual override `behind_nginx`).
2. **Firewall** -- автоопределение активного UFW для выбора стратегии (UFW / nftables / iptables).
3. **Ресурсы** -- CPU и RAM считываются через Ansible facts (`ansible_processor_vcpus`, `ansible_memtotal_mb`).
4. **Архитектура** -- amd64 / arm64 для загрузки правильного бинарника mtg.

### Контрактное

- **Входной контракт (CLI):** `ansible-playbook -i inventory.yml playbook.yml [--tags tag] [--limit host]`.
- **Контракт inventory:** YAML с обязательными группами `access_proxies` и `upstreams`, переменные `proxy_role`, `squid_port`, `ansible_host`.
- **Контракт шаблонов:** Jinja2 с доступом к Ansible facts и inventory variables.
- **Контракт клиентов:** JSON-файлы с полями `httpProxy` / `httpsProxy` (IP:port, credentials).

### Аналитическое

Ansible выбран как система развёртывания по следующим причинам:
- Agentless -- не требует установки агента на управляемые ноды (только SSH + Python).
- Декларативный YAML -- конфигурация читается как документация.
- Идемпотентность -- повторные запуски безопасны.
- Встроенная поддержка Jinja2 для шаблонизации конфигураций.
- Role-based организация для модульности.
- Параллельное выполнение для масштабируемости.
- Широкая экосистема модулей (apt, systemd, sysctl, ufw, iptables, nftables).

### Эмпирическое

Верификация:
- Полный deploy: `ansible-playbook -i inventory.yml playbook.yml` -> success.
- Идемпотентность: повторный deploy -> 0 changed.
- Selective deploy: `--tags crowdsec`, `--tags nginx`, `--tags mtproxy`.
- Limit deploy: `--limit access-01` -> только одна нода.
- Проверка конфигураций: `squid -k parse`, `nginx -t` при deploy.

### Негативное

| Риск | Митигация |
|---|---|
| SSH-ключ скомпрометирован | SSH-ключи хранятся отдельно; vault для секретов |
| Ошибка в шаблоне ломает конфигурацию | Validate при deploy (`squid -k parse`, `nginx -t`); intelligent reload/restart |
| Параллельный deploy с конфликтом | Конфигурации изолированы по нодам; shared config -- force:no |
| Сеть до ноды недоступна | Ansible retry; `--limit` для повторного deploy конкретной ноды |
| Пакет недоступен в репозитории | Retry с delay; alternative package managers (apt/dnf) |
