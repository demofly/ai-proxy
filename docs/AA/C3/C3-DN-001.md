# C3-DN-001 — DNS Resolver (Unbound)

## Ссылки

- Родительские требования C2: [C2-NF-003](../C2/C2-NF-003.md), [C2-NF-001](../C2/C2-NF-001.md)
- Дочерние артефакты C4: [C4-RL-004](../C4/C4-RL-004.md)

## Описание

Кеширующий DNS-резолвер (Unbound) развёртывается на каждой ноде инфраструктуры. Реализует каскадную архитектуру DNS-резолвинга: upstream-ноды резолвят через публичные DNS, access-ноды пересылают запросы на upstream unbound. Squid использует `dns_nameservers 127.0.0.1`.

### Каскадная архитектура

```
Публичные DNS (9.9.9.9, 8.8.8.8, 1.1.1.1)
          ↑
upstream: unbound (127.0.0.1:53 + ansible_host:53)
          ↑
access:   unbound (127.0.0.1:53) → forward → upstream unbound
          ↑
Squid:    dns_nameservers 127.0.0.1
```

### Ключевые решения

- **Access → upstream:** access-ноды НЕ резолвят через публичные DNS напрямую, а пересылают на upstream unbound. Это обеспечивает единую точку выхода в DNS и упрощает фильтрацию.
- **DNSSEC:** валидация включена по умолчанию (`auto-trust-anchor-file`).
- **systemd-resolved:** отключается автоматически при обнаружении (занимает порт 53).
- **Авторасчёт ресурсов:** потоки = CPU cores, кеш = RAM/32 (мин 16 MB).

### Интерфейсы

| Интерфейс | Описание |
|---|---|
| `127.0.0.1:53` | Локальный DNS для Squid и системы (все ноды) |
| `ansible_host:53` | Внешний DNS для access-нод (только upstream) |

### Конфигурация

| Параметр | Значение | Описание |
|---|---|---|
| `dns_resolver_port` | 53 | Порт unbound |
| `dns_resolver_dnssec` | true | DNSSEC-валидация |
| `dns_resolver_min_ttl` | 300 | Минимальный TTL (5 мин) |
| `dns_resolver_max_ttl` | 86400 | Максимальный TTL (24 ч) |
| `dns_resolver_neg_ttl` | 60 | Negative TTL |
| `dns_resolver_prefetch` | true | Обновление популярных записей до истечения TTL |

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | Unbound запущен и слушает | `systemctl is-active unbound` | active |
| 2 | Squid использует локальный DNS | `grep dns_nameservers /etc/squid/squid.conf` | `127.0.0.1` |
| 3 | DNS-резолвинг работает | `dig @127.0.0.1 anthropic.com` | NOERROR |
| 4 | Access → upstream forwarding | `dig @127.0.0.1 anthropic.com` на access-ноде | Ответ от upstream unbound |
| 5 | DNSSEC валидация | `dig @127.0.0.1 dnssec-failed.org` | SERVFAIL (невалидный DNSSEC) |

## Доказательство реализации

### Конструктивное

Реализовано в Ansible-роли `Servers/deploy/roles/dns_resolver/`:
- `defaults/main.yml` — параметры по умолчанию (порт, TTL, DNSSEC, prefetch).
- `tasks/main.yml` — установка unbound, авторасчёт ресурсов, конфигурация forwarders по `proxy_role`, отключение systemd-resolved.
- `templates/unbound.conf.j2` — шаблон конфигурации с условной логикой для access/upstream.

### Аналитическое

**Выбор Unbound:** лёгкий, безопасный (DNSSEC из коробки), хорошо подходит для forwarding/caching. Альтернативы: BIND (избыточен для caching-only), dnsmasq (слабее DNSSEC).

**Каскадная архитектура:** access-ноды не выходят в публичный DNS напрямую — весь DNS-трафик проходит через upstream, что обеспечивает единую точку контроля и снижает DNS-утечки.

### Негативное

| Риск | Митигация |
|---|---|
| Upstream unbound недоступен | Unbound пытает все forwarders; при полном отказе upstream — Squid fallback на DNS timeout |
| DNS-спуфинг | DNSSEC-валидация включена по умолчанию |
| Порт 53 занят systemd-resolved | Автоматическое отключение stub listener в роли |
