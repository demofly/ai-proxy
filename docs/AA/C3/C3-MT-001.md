# C3-MT-001 -- MTProxy (mtg)

## Ссылки

- Родительские требования C2:
  - [C2-FR-006](../C2/C2-FR-006.md) -- MTProxy для Telegram
- Дочерние требования C4:
  - [C4-RL-003](../C4/C4-RL-003.md) -- Ansible role: mtproxy

## Описание

MTProxy (mtg) -- компонент Telegram-проксирования, развёрнутый на upstream-нодах. Реализует протокол MTProto Proxy с fake-TLS маскировкой для обхода DPI (Deep Packet Inspection). Использует бинарный файл `mtg` v2 от проекта [github.com/9seconds/mtg](https://github.com/9seconds/mtg).

### Принцип работы

mtg принимает TLS-подключения от Telegram-клиентов, расшифровывает fake-TLS обёртку, извлекает MTProto-трафик и проксирует его к серверам Telegram DC (Data Center). Для внешнего наблюдателя (DPI) трафик выглядит как обычное HTTPS-подключение к популярному домену (по умолчанию `example.gov.ru`).

### Архитектура подключения

Telegram-клиент подключается через цепочку:
1. Клиент -> access-нода (порт 443, TLS с SNI=`mtproxy_fake_tls_domain`).
2. nginx SNI Router на access-ноде маршрутизирует по SNI на upstream MTProxy.
3. mtg на upstream-ноде (порт 2083) обрабатывает fake-TLS и проксирует к Telegram DC.

### Конфигурация

| Параметр | Значение по умолчанию | Описание |
|---|---|---|
| `mtproxy_enabled` | false | Включение/отключение MTProxy |
| `mtproxy_version` | 2.1.8 | Версия mtg |
| `mtproxy_local_port` | 2083 | Локальный порт daemon |
| `mtproxy_secret` | "" (автогенерация) | Hex-секрет для fake-TLS |
| `mtproxy_fake_tls_domain` | example.gov.ru | Домен для fake-TLS маскировки |
| `mtproxy_hostname` | "" | Публичный hostname для SNI-маршрутизации |
| `mtproxy_users` | [] | Реестр пользователей (для администратора) |

### Конфигурационные файлы

| Файл | Описание |
|---|---|
| `/etc/mtproxy/mtg.toml` | Основная конфигурация mtg v2 (secret, bind, fake-tls) |
| `/etc/mtproxy/users.txt` | Реестр пользователей (имя, telegram, дата доступа, заметка) |
| `/etc/systemd/system/mtg.service` | Systemd unit (User=mtproxy, MemoryMax=64M, LimitNOFILE=4096) |

### Управление секретом

1. Если `mtproxy_secret` задан в inventory -- используется он.
2. Если на ноде уже есть `mtg.toml` -- секрет читается из существующей конфигурации.
3. Если есть legacy `mtg.env` (миграция с v1) -- секрет читается оттуда, env-файл удаляется.
4. Иначе -- генерируется новый секрет командой `mtg generate-secret --hex <domain>`.

## Механизм реализации

### Механизм fake-TLS маскировки

1. Telegram-клиент получает ссылку `tg://proxy?server=...&port=443&secret=ee...` (secret начинается с `ee` = fake-TLS режим).
2. Клиент инициирует TLS-подключение к access-ноде на порт 443.
3. TLS ClientHello содержит SNI = `mtproxy_fake_tls_domain` (например, `example.gov.ru`).
4. nginx SNI Router на access-ноде маршрутизирует TCP-соединение на upstream MTProxy (порт 2083).
5. mtg принимает соединение, расшифровывает fake-TLS обёртку с использованием shared secret.
6. Извлечённый MTProto-трафик проксируется к Telegram DC.
7. Для DPI трафик неотличим от обычного HTTPS к example.gov.ru.

### Механизм развёртывания

1. Создаётся системный пользователь `mtproxy` (nologin).
2. Определяется архитектура системы (amd64 / arm64).
3. Скачивается и устанавливается бинарник mtg из GitHub Releases.
4. Создаётся директория конфигурации `/etc/mtproxy/` (owner: mtproxy, mode: 0750).
5. Разрешается или генерируется секрет (см. "Управление секретом").
6. Развёртывается `mtg.toml` и `users.txt` из шаблонов.
7. Создаётся systemd unit `mtg.service`.
8. Настраивается firewall (порт 2083 -- только от IP access-прокси).
9. Активируется сервис mtg.

### Механизм выбора firewall для MTProxy

Аналогично upstream Squid -- мультиплатформенный подход:

1. **Ubuntu** -- UFW rules: allow порт 2083 от IP access-прокси.
2. **Debian** -- nftables: отдельная конфигурация `/etc/nftables.d/mtproxy.conf`.
3. **RHEL/CentOS** -- iptables: собственная цепочка `MTPROXY` с allow от access IP + drop остальных.

### Контрактное

- **Входной контракт:** fake-TLS подключение на порт 2083 с shared secret (ee-prefix hex). Протокол: MTProto Proxy v2.
- **Выходной контракт:** MTProto-подключение к Telegram DC серверам.
- **Контракт с nginx:** TCP-поток от SNI Router (proxy_pass) -- без модификации.
- **Контракт пользователей:** `tg://proxy?server={access_ip}&port=443&secret={secret}` -- ссылка для Telegram-клиента.

### Аналитическое

mtg выбран как реализация MTProxy по следующим причинам:
- Go-бинарник (single binary) -- простая установка, нет зависимостей.
- Нативная поддержка fake-TLS (обход DPI) -- основная цель проекта.
- TOML-конфигурация (v2) -- лаконичная и читаемая.
- Минимальный footprint: 64 MB RAM, 4096 файловых дескрипторов.
- Активный проект с регулярными релизами.
- Поддержка ARM64 (серверы Hetzner ARM).

### Эмпирическое

Верификация:
- `systemctl is-active mtg` -> active.
- `ss -tlnp :2083` -> mtg слушает.
- Подключение Telegram-клиента через прокси-ссылку -> Connected.
- Отправка сообщения через прокси -> успех.

### Негативное

| Риск | Митигация |
|---|---|
| DPI обнаруживает fake-TLS | Использование популярного домена (example.gov.ru); SNI неотличим от легитимного HTTPS |
| Утечка секрета | Ansible `no_log: true` для задач с секретом; файл mode 0640, owner mtproxy |
| Неавторизованный доступ к порту 2083 | Firewall: только IP access-прокси; секрет = криптографическая авторизация |
| mtg бинарник недоступен (GitHub) | Бинарник загружается только при первом deploy; при наличии -- пропускается |
| Исчерпание памяти | systemd MemoryMax=64M; OOM killer при превышении |
