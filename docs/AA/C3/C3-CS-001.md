## Описание

CrowdSec IPS (Intrusion Prevention System) -- компонент активной защиты, развёрнутый на всех нодах системы. Состоит из двух подсистем:

1. **CrowdSec Security Engine (агент)** -- анализирует логи (Squid access.log, SSH auth.log, journald) на основе сценариев обнаружения атак. Использует коллективную threat intelligence от сообщества CrowdSec (70k+ серверов) через CAPI (Central API).

2. **CrowdSec Firewall Bouncer (nftables)** -- применяет решения агента, блокируя вредоносные IP-адреса через nftables rules. Автоматическое добавление и удаление ban-правил.

### Коллекции и сценарии обнаружения

| Коллекция | Описание | Источник логов |
|---|---|---|
| `crowdsecurity/linux` | SSH brute-force, failed logins, invalid users | auth.log, journald (ssh.service) |
| `crowdsecurity/nginx` | HTTP атаки, сканеры, боты | -- (применяется при наличии nginx) |
| `crowdsecurity/http-cve` | Эксплуатация известных CVE | HTTP-логи |
| `crowdsecurity/whitelist-good-actors` | Белый список легитимных сканеров (Googlebot и т.д.) | -- |

### Источники логов (Acquisition)

Конфигурация определяется в `acquis.yaml.j2`:

1. **Squid access.log** -- только на access-прокси с включённым журналированием (`proxy_role == 'access' and enable_logging`). Тип: `squid`.
2. **System auth.log** (`/var/log/auth.log`) -- на всех нодах. Тип: `syslog`.
3. **journald** (ssh.service) -- дублирование SSH-логов через systemd journal для надёжности. Тип: `syslog`.

### Ресурсные ограничения

| Компонент | Лимит памяти | Настройка |
|---|---|---|
| CrowdSec agent | 512 MB | systemd MemoryMax override |
| Firewall bouncer | 128 MB | systemd MemoryMax override |

Лимиты настраиваются через systemd override файлы в `/etc/systemd/system/{service}.service.d/memory-limit.conf`.

## Механизм реализации

### Механизм обнаружения и блокировки

1. CrowdSec агент читает настроенные источники логов (файлы + journald).
2. Парсеры преобразуют строки логов в структурированные события.
3. Сценарии обнаружения (collections) анализируют события на паттерны атак (brute-force, сканирование, эксплуатация CVE).
4. При срабатывании сценария -- агент создаёт decision (IP ban на определённый срок).
5. Firewall bouncer получает decision и добавляет nftables rule для блокировки IP.
6. По истечении срока ban -- nftables rule автоматически удаляется.

### Механизм коллективной защиты (CAPI)

1. CrowdSec агент регистрируется в CAPI при первом запуске.
2. Агент периодически отправляет анонимизированные alert-данные в CAPI.
3. CAPI агрегирует данные от 70k+ серверов и формирует blocklist.
4. Агент получает обновлённый blocklist и применяет превентивные ban через bouncer.
5. Результат: IP, замеченные в атаках на другие серверы сообщества, блокируются превентивно.

### Механизм развёртывания

1. Ansible добавляет GPG-ключ и apt-репозиторий CrowdSec.
2. Устанавливаются пакеты `crowdsec` и `crowdsec-firewall-bouncer-nftables` (без wizard: `CROWDSEC_PACKAGE_DISABLE_WIZARD=1`).
3. Обновляется hub index (`cscli hub update`) с retry (3 попытки, задержка 10 секунд).
4. Устанавливаются коллекции (`cscli collections install`).
5. Развёртывается `acquis.yaml` из шаблона.
6. Настраиваются systemd memory limits.
7. Активируются сервисы `crowdsec` и `crowdsec-firewall-bouncer`.

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | CrowdSec агент запущен | systemctl is-active crowdsec | active |
| 2 | Firewall bouncer запущен | systemctl is-active crowdsec-firewall-bouncer | active |
| 3 | Коллекции установлены | cscli collections list -> linux, nginx, http-cve, whitelist-good-actors | installed |
| 4 | Acquisition настроен | cat /etc/crowdsec/acquis.yaml -> корректные источники | pass |
| 5 | CAPI подключён | cscli capi status | registered |
| 6 | Memory limits настроены | systemctl show crowdsec -> MemoryMax=200M | pass |
| 7 | SSH brute-force обнаруживается | Тестовые failed login -> cscli decisions list | ban IP |
| 8 | nftables rules обновляются | nft list ruleset -> crowdsec ban rules | pass |

### Контрактное

- **Входной контракт (logfiles):** текстовые файлы логов в стандартных форматах (squid, syslog) + journald (systemd unit filter).
- **Выходной контракт (nftables):** добавление/удаление nftables rules с IP-адресами в таблице crowdsec.
- **Контракт CAPI:** HTTPS REST API -- отправка alerts, получение decisions (blocklist).

### Аналитическое

CrowdSec выбран как IPS по следующим причинам:
- Open-source с коллективной threat intelligence (70k+ серверов).
- Лёгкий агент с минимальным footprint (200 MB RAM).
- Нативная интеграция с nftables (firewall bouncer) -- без промежуточных слоёв.
- Поддержка множества форматов логов (squid, syslog, nginx, journald).
- Декларативная конфигурация acquisition через YAML.
- Автоматическое управление ban/unban с TTL -- не требует ручного вмешательства.

### Эмпирическое

Верификация:
- `systemctl is-active crowdsec crowdsec-firewall-bouncer` -> active.
- `cscli collections list` -> все 4 коллекции installed.
- `cscli capi status` -> registered.
- Тестовый brute-force SSH -> `cscli decisions list` -> ban IP.
- `nft list ruleset` -> crowdsec ban rules присутствуют.

### Негативное

| Риск | Митигация |
|---|---|
| Ложное срабатывание (блокировка легитимного IP) | Коллекция `whitelist-good-actors` (Googlebot и т.д.); `cscli decisions delete` для ручного разбана |
| CAPI недоступен | Агент работает автономно с локальными сценариями; CAPI -- дополнительный слой |
| Исчерпание памяти агентом | systemd MemoryMax=200MB; OOM killer при превышении |
| CrowdSec пакет недоступен | Retry 3 раза с задержкой 10 секунд; dpkg --configure -a при broken state |
| Конфликт nftables с другими firewall-правилами | CrowdSec bouncer использует собственную таблицу nftables |
