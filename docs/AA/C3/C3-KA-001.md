## Описание

Keepalived VRRP -- компонент отказоустойчивости access tier, обеспечивающий автоматический failover виртуального IP-адреса (VIP) между access-нодами. Использует протокол VRRP (Virtual Router Redundancy Protocol) для определения мастер-ноды и переключения при отказе.

### Архитектура

- **MASTER / BACKUP** -- access-ноды имеют роли с приоритетами (MASTER: priority=100, BACKUP: priority=90).
- **VIP** -- виртуальный IP-адрес (`keepalived_vip`), который мигрирует между нодами. Клиенты подключаются к VIP.
- **Health check** -- скрипт `check_squid`, который проверяет работоспособность Squid через `killall -0 squid`.
- **VRID** -- Virtual Router ID (по умолчанию 51), уникальный в пределах сегмента L2.

### Конфигурация

| Параметр | Значение | Описание |
|---|---|---|
| `keepalived_state` | MASTER / BACKUP | Роль ноды |
| `keepalived_priority` | 100 (MASTER) / 90 (BACKUP) | Приоритет VRRP |
| `keepalived_vip` | IP-адрес | Виртуальный IP |
| `keepalived_vip_prefix` | 24 | Маска подсети VIP |
| `keepalived_vrid` | 51 | Virtual Router ID |
| `keepalived_interface` | auto / manual | Сетевой интерфейс (автоопределение через `ansible_default_ipv4.interface`) |
| `keepalived_auth_pass` | строка | Пароль аутентификации VRRP |
| `keepalived_router_id` | AI_PROXY_RU | Идентификатор маршрутизатора |
| `advert_int` | 1 секунда | Интервал VRRP-анонсов |

### Health check: check_squid

```
vrrp_script check_squid {
    script "/usr/bin/killall -0 squid"
    interval 2        # Проверка каждые 2 секунды
    weight -20         # При неуспехе: priority -= 20
    fall 3             # 3 последовательных неуспеха -> FAULT
    rise 2             # 2 последовательных успеха -> OK
}
```

Механизм: `killall -0` проверяет существование процесса без отправки сигнала. При отказе Squid приоритет ноды снижается на 20, что приводит к переключению VIP на BACKUP-ноду (если её приоритет выше).

### Условия активации

Keepalived разворачивается только при выполнении всех условий:
- `proxy_role == 'access'`
- `enable_keepalived == true`
- `keepalived_vip is defined`

Если `keepalived_vip` не задан -- Ansible выводит информационное сообщение и пропускает развёртывание.

## Механизм реализации

### Механизм failover

1. MASTER отправляет VRRP-анонсы каждую секунду (`advert_int 1`) на multicast 224.0.0.18.
2. BACKUP получает анонсы и подтверждает статус MASTER.
3. Health check `check_squid` проверяет процесс Squid каждые 2 секунды.
4. Если Squid падает -- 3 последовательных неуспеха (`fall 3`) -> приоритет MASTER снижается на 20 (100 -> 80).
5. BACKUP (priority=90) обнаруживает, что его приоритет выше -> принимает роль MASTER.
6. VIP мигрирует на BACKUP-ноду (gratuitous ARP).
7. Клиенты переключаются на новую ноду прозрачно (VIP не меняется).

### Механизм восстановления

1. Squid перезапускается на бывшей MASTER-ноде.
2. 2 последовательных успеха (`rise 2`) -> приоритет восстанавливается до 100.
3. MASTER (priority=100) возвращает роль (preemption по умолчанию включён).
4. VIP мигрирует обратно на первичную ноду.

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | VIP доступен | ping keepalived_vip -> ответ | pass |
| 2 | Failover при остановке Squid | systemctl stop squid -> VIP мигрирует на BACKUP | < 10 секунд |
| 3 | Восстановление после рестарта Squid | systemctl start squid -> VIP возвращается на MASTER | < 10 секунд |
| 4 | VRRP-анонсы отправляются | tcpdump vrrp -> пакеты каждую секунду | pass |
| 5 | ip_nonlocal_bind включён | sysctl net.ipv4.ip_nonlocal_bind == 1 | pass |
| 6 | Конфигурация валидна | keepalived --check -> exit code 0 | pass |

### Контрактное

- **Контракт VRRP:** MASTER отправляет анонсы на 224.0.0.18 (VRID, priority, auth). BACKUP принимает роль при прекращении анонсов или снижении приоритета MASTER.
- **Контракт health check:** `killall -0 squid` -> exit code 0 = процесс жив, != 0 = процесс мёртв.
- **Контракт VIP:** IP-адрес добавляется/удаляется на сетевом интерфейсе; gratuitous ARP обновляет ARP-таблицы в сегменте.

### Аналитическое

Keepalived с VRRP выбран как механизм HA по следующим причинам:
- Стандартный протокол (RFC 5798) для failover виртуального IP.
- Минимальный overhead: один процесс, один multicast-поток.
- Не требует внешней инфраструктуры (в отличие от DNS failover, load balancer).
- Мгновенное переключение (< 10 секунд) -- определяется `advert_int` + `fall` * `interval`.
- Интеграция с health check Squid -- failover привязан к здоровью прокси-сервиса.

### Эмпирическое

Верификация:
- Функциональная: `systemctl stop squid` на MASTER -> VIP мигрирует на BACKUP (проверка через `ip addr show`).
- Восстановление: `systemctl start squid` -> VIP возвращается.
- Сетевая: `tcpdump -i any vrrp` -> VRRP-анонсы видны.
- Клиентская: curl через VIP работает при failover.

### Негативное

| Риск | Митигация |
|---|---|
| Split-brain (оба MASTER) | Аутентификация VRRP (auth_pass); единый VRID в L2-сегменте |
| Ложный failover (временный сбой Squid) | `fall 3` -- 3 последовательных неуспеха перед переключением |
| Ложное восстановление (нестабильный Squid) | `rise 2` -- 2 последовательных успеха перед возвратом |
| VIP не привязывается | `ip_nonlocal_bind = 1` через sysctl |
| VRRP blocked firewall | VRRP (IP protocol 112) должен быть разрешён |
