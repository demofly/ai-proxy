# ADR-000003 — nginx SNI-маршрутизация для co-deployment

## Статус

Implemented

## Контекст

На upstream- и access-нодах на одном IP-адресе сосуществуют несколько сервисов, требующих TLS на порту 443: Squid-прокси (CONNECT-туннелирование) и MTProxy (fake-TLS для Telegram). На одном IP-адресе может быть привязан к порту 443 только один сервис.

Необходимо решение для бесконфликтного разделения порта 443 между сервисами без расшифровки TLS-трафика (TLS termination выполняется конечными сервисами).

**Важное ограничение:** nginx `ssl_preread` работает только с TLS-трафиком (извлекает SNI из ClientHello). Plain HTTP-соединения (в т.ч. Squid CONNECT) не содержат TLS ClientHello и отбрасываются ssl_preread. Это означает, что Squid не может работать за nginx SNI Router на upstream-нодах — только сервисы с TLS (MTProxy fake-TLS).

## Решение

Принята **SNI-маршрутизация через nginx stream модуль** с разной топологией на access и upstream:

![Последовательность SNI-маршрутизации: TLS ClientHello → ssl_preread → map → backend](diagrams/ADR-000003-sni-routing.svg)

> Исходник: [diagrams/ADR-000003-sni-routing.puml](diagrams/ADR-000003-sni-routing.puml)

- nginx слушает порт 443 в режиме `stream` (L4)
- Директива `ssl_preread on` извлекает SNI (Server Name Indication) из TLS ClientHello
- Переменная `$ssl_preread_server_name` определяет целевой backend
- `map` сопоставляет SNI-домены с upstream-бэкендами (через include `sni-map.d/*.conf`)
- Трафик проксируется на соответствующий backend без расшифровки TLS
- `proxy_protocol on` передаёт реальный IP клиента бэкенду

**Access-ноды** — nginx:443 маршрутизирует:
- Весь HTTPS-трафик пользователей → Squid на `127.0.0.1:3128` (с PROXY protocol, `require-proxy-header`)
- MTProxy fake-TLS → локальный MTProxy daemon (если включён)

**Upstream-ноды** — nginx:443 маршрутизирует **только MTProxy**:
- MTProxy fake-TLS → mtg на `127.0.0.1:2083`
- Squid-трафик **не проходит** через nginx — Squid слушает на порту 80 напрямую ([ADR-000002](ADR-000002.md))
- `fallback_squid` (`127.0.0.1:3128`) — резервный бэкенд для немаршрутизированного TLS-трафика

```nginx
# AI-GENERATED — NOT REVIEWED: SECTION START
# Upstream-нода: nginx :443 — только MTProxy SNI routing
stream {
    map $ssl_preread_server_name $sni_backend {
        include /etc/nginx/stream.d/sni-map.d/*.conf;
        default fallback_squid;
    }

    upstream fallback_squid {
        server 127.0.0.1:3128;
    }

    server {
        listen 443;
        listen [::]:443;
        ssl_preread on;
        proxy_pass $sni_backend;
        proxy_protocol on;
        proxy_connect_timeout 10s;
        proxy_timeout 1800s;
        proxy_socket_keepalive on;
    }
}
# AI-GENERATED — NOT REVIEWED: SECTION END
```

## Альтернативы

### Отдельные IP-адреса (отклонено)

Выделить каждому проекту отдельный IP-адрес.

**Причина отклонения:** дополнительные затраты на IP-адреса; не все хостинг-провайдеры предоставляют несколько IP на один сервер; усложняет DNS-конфигурацию.

### Разные порты (отклонено)

Использовать нестандартные порты (8443, 9443) для разных проектов.

**Причина отклонения:** нестандартные порты могут блокироваться корпоративными фаерволами; ухудшает маскировку трафика (MTProxy fake-TLS рассчитан на порт 443).

### HAProxy SNI routing (отклонено)

Использовать HAProxy вместо nginx для SNI-маршрутизации.

**Причина отклонения:** дополнительная зависимость. nginx уже используется как стандартный компонент в стеке. HAProxy избыточен для L4-маршрутизации по SNI.

## Последствия

### Положительные

- Бесконфликтное разделение порта 443 между проектами
- Отсутствие расшифровки TLS (end-to-end encryption сохраняется)
- Минимальная добавленная латентность (L4 proxying)
- Лёгкое добавление новых проектов (map + upstream)

### Отрицательные

- Дополнительный компонент (nginx) на upstream-нодах
- Все TLS-сервисы на ноде зависят от nginx (single point of failure для порта 443)
- SNI передаётся в открытом виде (потенциально видно DPI)
- Squid-трафик на upstream не может проходить через SNI Router (plain HTTP несовместим с ssl_preread) — требуется отдельный порт (80)

### Риски

- Encrypted Client Hello (ECH) в будущем может скрыть SNI, что сделает маршрутизацию невозможной без расшифровки
- Ошибка в конфигурации nginx может нарушить маршрутизацию всех проектов
