# ADR-000002 — Squid как прокси-движок

## Статус

Implemented

## Контекст

Для реализации двухуровневой архитектуры ([ADR-000001](ADR-000001.md)) необходим HTTP/HTTPS прокси-сервер, обеспечивающий:

- CONNECT-туннелирование (TLS passthrough без расшифровки)
- Аутентификацию через Basic Auth
- Гибкую фильтрацию доменов (ACL: exact match, regex)
- Кеширование HTTP-ответов
- Каскадирование на upstream-прокси (cache_peer) с балансировкой
- SMP-параллелизм (использование нескольких CPU)
- Зрелость, стабильность, активное community

## Решение

Выбран **Squid** (версия 6.x) в качестве прокси-движка для обоих уровней:

**Access tier:**
- `http_port 127.0.0.1:3128 require-proxy-header` — за nginx SNI Router ([ADR-000003](ADR-000003.md)), реальный IP клиента через PROXY protocol
- `auth_param basic program /usr/lib/squid/basic_ncsa_auth` — аутентификация
- `acl allowed_domains dstdomain` + `dstdom_regex` — фильтрация доменов
- `cache_dir ufs` — дисковый кэш
- `cache_peer ... parent 80 0 no-query userhash` — каскадирование на upstream (plain HTTP)
- `workers N` — SMP-параллелизм
- `forwarded_for off` — скрытие X-Forwarded-For (анонимизация)

**Upstream tier:**
- `http_port 80` — порт прокси (напрямую на внешнем интерфейсе, без nginx)
- `forwarded_for off` — удаление X-Forwarded-For (анонимизация)
- `via off` — удаление Via заголовка
- Без аутентификации, без кеширования, без логирования
- ACL: доступ только с IP access-прокси (UFW + Squid ACL)

## Альтернативы

### nginx proxy_pass (отклонено)

Использовать nginx с директивой `proxy_pass` для проксирования.

**Причина отклонения:** nginx не поддерживает CONNECT-метод нативно (требуется сторонний модуль `ngx_http_proxy_connect_module`). Нет встроенной поддержки каскадирования (cache_peer), Basic Auth через файл паролей требует дополнительного модуля. Кеширование ограничено.

### HAProxy (отклонено)

Использовать HAProxy как L7-прокси.

**Причина отклонения:** HAProxy ориентирован на балансировку нагрузки, а не на forward proxy. Не поддерживает CONNECT-туннелирование, HTTP-кеширование, фильтрацию по доменам. Неподходящий инструмент для задачи.

### Envoy (отклонено)

Использовать Envoy Proxy.

**Причина отклонения:** избыточная сложность для задачи forward proxy. Конфигурация через YAML/xDS значительно сложнее Squid. Нет нативной поддержки ncsa_auth. Ориентирован на service mesh, а не на корпоративный forward proxy.

### Tinyproxy (отклонено)

Использовать Tinyproxy — лёгкий HTTP-прокси.

**Причина отклонения:** не поддерживает каскадирование (cache_peer), SMP, продвинутые ACL. Минимальный функционал недостаточен для требований проекта.

## Последствия

### Положительные

- Зрелое решение (30+ лет разработки), активное community
- Богатый набор функций: ACL, кеширование, аутентификация, каскадирование — из коробки
- Доступен в стандартных репозиториях Ubuntu/Debian (apt install squid)
- SMP-параллелизм для эффективного использования многоядерных серверов
- Обширная документация и база знаний

### Отрицательные

- Конфигурация squid.conf может быть сложной для новичков
- Размер памяти: Squid потребляет значительный объём RAM при кешировании
- Устаревший формат конфигурации (не YAML/JSON)

### Риски

- Squid 6.x — относительно новая мажорная версия; возможны баги
- Зависимость от единственного прокси-движка на обоих уровнях
