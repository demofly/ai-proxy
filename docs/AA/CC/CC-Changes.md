# CC-Changes — Журнал управления изменениями

## Категории изменений (TOGAF Phase H)

| Категория | Описание | Примеры | Требует ADR |
|---|---|---|---|
| **Simplification** | Упрощение без изменения архитектуры | Рефакторинг шаблонов, оптимизация конфигураций | Нет |
| **Incremental** | Добавление функциональности в рамках существующей архитектуры | Новые роли Ansible, расширение whitelist, добавление документации | По необходимости |
| **Re-Architecting** | Существенное изменение архитектуры | Смена прокси-сервера, добавление уровня, смена схемы аутентификации | Да |

## Процедура управления изменениями

### Инициация изменения

1. **Идентификация потребности** — выявление необходимости изменения (новое требование, gap, инцидент, обратная связь).
2. **Классификация** — определение категории (Simplification / Incremental / Re-Architecting).
3. **Регистрация CR** — запись в журнал с уникальным CR-ID.

### Оценка и согласование

1. **Анализ влияния** — определение зоны влияния.
2. **Оценка рисков** — проверка по [CC-Risks.md](CC-Risks.md).
3. **Согласование** — с ролью Accountable по [CC-RACI.md](CC-RACI.md).
4. **ADR** — при категории Re-Architecting или значимом Incremental.

### Реализация

1. **Выполнение** — реализация изменения.
2. **Верификация** — проверка результата.
3. **Актуализация документации** — сквозная актуализация.

### Статусы CR

| Статус | Описание |
|---|---|
| **Proposed** | CR зарегистрирован, ожидает оценки |
| **Approved** | CR одобрен, ожидает реализации |
| **In Progress** | CR в процессе реализации |
| **Completed** | CR реализован и верифицирован |
| **Rejected** | CR отклонён с обоснованием |
