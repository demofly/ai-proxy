# C4-RL-003 -- Роль MTProxy (roles/mtproxy/)

## Ссылки

- Родительские требования C3: [C3-MT-001](../C3/C3-MT-001.md)
- Связанные C4: [C4-PB-001](C4-PB-001.md) (включает роль при upstream + mtproxy_enabled), [C4-RL-002](C4-RL-002.md) (nginx SNI routing для MTProxy на access-узлах)

## Описание

Директория `Servers/deploy/roles/mtproxy/` -- Ansible-роль для развёртывания Telegram MTProxy (mtg v2) на upstream-узлах. Обеспечивает скачивание и установку бинарника mtg, генерацию или восстановление секрета, развёртывание конфигурации, настройку systemd-сервиса и firewall.

### Состав роли

| Файл | Описание |
|---|---|
| `tasks/main.yml` | 285 строк: установка mtg, управление секретами, деплой конфигурации, systemd, firewall (UFW/nftables/iptables) |
| `handlers/main.yml` | 3 handler: reload systemd, restart mtg, restart nftables |
| `defaults/main.yml` | Переменные по умолчанию: enabled, version, port, secret, hostname, users |
| `templates/mtg.toml.j2` | TOML-конфигурация mtg v2 (secret, bind-to, fake-tls) |
| `templates/mtg.env.j2` | Legacy env-файл (v1 формат, для обратной совместимости) |
| `templates/users.txt.j2` | Реестр пользователей MTProxy (административный, не используется mtg) |
| `templates/nftables-mtproxy.conf.j2` | nftables-правила для MTProxy на Debian |
| `meta/main.yml` | Метаданные роли |

## Ключевые механизмы

### Управление секретами (3 стратегии)

Механизм приоритезированного разрешения секрета:

1. **Явный секрет из инвентори** -- если `mtproxy_secret != ""`, используется он
2. **Существующий из TOML** -- если mtg.toml существует, секрет извлекается из него (`grep '^secret'`)
3. **Существующий из legacy env** -- если mtg.env существует (миграция v1 -> v2), секрет извлекается из него
4. **Генерация нового** -- `mtg generate-secret --hex {fake_tls_domain}` при первом деплое

Все операции с секретами помечены `no_log: true`.

```yaml
mtproxy_secret_resolved: >-
  {%- if mtproxy_secret != "" -%}
    {{ mtproxy_secret }}
  {%- elif mtg_toml_file.stat.exists -%}
    {{ mtg_existing_secret_toml.stdout }}
  {%- elif mtg_env_file.stat.exists -%}
    {{ mtg_existing_secret_env.stdout }}
  {%- else -%}
    {{ mtg_generated_secret.stdout }}
  {%- endif -%}
```

### Миграция v1 -> v2

Роль поддерживает обратную совместимость с форматом v1 (env-файл):

- Проверяется наличие legacy mtg.env
- Если найден -- секрет извлекается, конфиг конвертируется в TOML
- Legacy mtg.env удаляется после успешной миграции

### Multi-arch binary install

Бинарник mtg скачивается с GitHub releases с автоопределением архитектуры:

```yaml
mtg_arch: "{{ 'arm64' if ansible_architecture == 'aarch64' else 'amd64' }}"
```

Установка выполняется только если бинарник отсутствует (идемпотентность).

### Multi-OS firewall

Три стратегии firewall для upstream-узлов:

| OS | Стратегия | Механизм |
|---|---|---|
| **Ubuntu** | UFW | `ufw allow from {access_ip} to any port {mtproxy_port} proto tcp` |
| **Debian** | nftables | Отдельная таблица `inet mtproxy` с drop unauthorized |
| **RHEL/CentOS** | iptables | Кастомная цепочка `MTPROXY` с jump из INPUT |

Все стратегии разрешают MTProxy только от IP-адресов access-прокси.

### Реестр пользователей (users.txt)

Административный файл (не используется mtg): содержит реестр пользователей с Telegram-аккаунтами, датами выдачи доступа и заметками. Необходим для процедуры ротации секрета:

1. Сменить `mtproxy_secret` в inventory
2. Удалить отозванного пользователя из `mtproxy_users`
3. Применить playbook
4. Разослать новую ссылку оставшимся пользователям

## Переменные

### defaults/main.yml

| Переменная | Тип | Значение по умолчанию | Описание |
|---|---|---|---|
| `mtproxy_enabled` | bool | `false` | Включение MTProxy |
| `mtproxy_version` | string | `2.1.8` | Версия mtg |
| `mtproxy_local_port` | int | `2083` | Локальный порт mtg daemon |
| `mtproxy_secret` | string | `""` | Hex-секрет (пустая строка = автогенерация) |
| `mtproxy_hostname` | string | `""` | Публичный домен для SNI routing |
| `mtproxy_users` | list | 1 default user | Реестр пользователей |

### Из inventory (global vars)

| Переменная | Тип | Описание |
|---|---|---|
| `mtproxy_fake_tls_domain` | string | Fake-TLS домен для DPI-маскировки |
| `mtproxy_max_connections` | int | Макс. соединений per IP |

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | mtg daemon запущен | `systemctl is-active mtg` | active |
| 2 | mtg слушает порт | `ss -tlnp sport = :2083` | mtg на 0.0.0.0:2083 |
| 3 | Секрет сохранён | `/etc/mtproxy/mtg.toml` содержит secret | Не пустой |
| 4 | Firewall настроен | Проверка UFW/nftables/iptables | MTProxy доступен только от access-прокси |
| 5 | Users.txt актуален | `/etc/mtproxy/users.txt` | Содержит текущий реестр |
| 6 | Telegram подключение | tg://proxy ссылка | Успешное подключение |
| 7 | Memory limit | `systemctl show mtg -p MemoryMax` | <= 64M |

## Доказательство реализации

### Конструктивное

Роль реализует полный жизненный цикл MTProxy: от установки бинарника до настройки firewall. Механизм управления секретами обеспечивает три стратегии разрешения (явный, существующий, генерируемый) с поддержкой миграции v1->v2. Multi-OS firewall адаптируется к дистрибутиву.

### Трассировочное

| C3 | C4 |
|---|---|
| [C3-MT-001](../C3/C3-MT-001.md) | C4-RL-003 -- развёртывание MTProxy |

### Эмпирическое

- Проверка сервиса: `systemctl status mtg`
- Проверка подключения: Telegram клиент с tg://proxy ссылкой
- Проверка секрета: `grep secret /etc/mtproxy/mtg.toml`
- Проверка firewall: попытка подключения к порту 2083 от стороннего IP

### Негативное

- **Утечка секрета** -- `no_log: true` на всех операциях с секретами
- **Потеря секрета при переустановке** -- чтение существующего из конфига перед генерацией
- **Несовместимость v1/v2** -- миграция mtg.env -> mtg.toml с сохранением секрета
- **Несанкционированный доступ** -- firewall ограничивает доступ к порту MTProxy
- **OOM** -- systemd MemoryMax=64M, LimitNOFILE=4096
