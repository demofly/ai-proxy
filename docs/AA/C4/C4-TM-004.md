# C4-TM-004 -- Шаблон nftables Upstream (nftables-upstream.conf.j2)

## Ссылки

- Родительские требования C3: [C3-SU-001](../C3/C3-SU-001.md)
- Связанные C4: [C4-PB-001](C4-PB-001.md) (деплоит шаблон), [C4-TM-002](C4-TM-002.md) (upstream Squid, защищаемый этим firewall)

## Описание

Файл `Servers/deploy/templates/nftables-upstream.conf.j2` (~42 строки) -- Jinja2-шаблон правил межсетевого экрана nftables для upstream-узлов на Debian. Генерирует `/etc/nftables.conf`. Обеспечивает политику default drop с whitelist-доступом: SSH, Squid (только от access-прокси), MTProxy TLS port 443 (условно, при `mtproxy_enabled`), ICMP, established/related.

## Структура файла

| Секция | Строки (прибл.) | Описание |
|---|---|---|
| **Shebang и заголовок** | 1--4 | `#!/usr/sbin/nft -f`, комментарии с ansible_managed |
| **flush ruleset** | 6 | Очистка всех правил перед применением |
| **table inet filter** | 8--41 | Таблица фильтрации для IPv4+IPv6 |
| **chain input** | 9--32 | Входящий трафик: policy drop, whitelist-правила |
| **chain forward** | 34--35 | Транзитный трафик: policy drop |
| **chain output** | 37--39 | Исходящий трафик: policy accept |

## Ключевые механизмы

### Политика default drop

Базовая политика `policy drop` для цепочек input и forward. Исходящий трафик разрешён полностью (`policy accept`).

### Динамический whitelist для Squid

```
{% for host in groups['access_proxies'] %}
ip saddr {{ hostvars[host]['ansible_host'] }} tcp dport {{ squid_port }} accept
{% endfor %}
```

Правила генерируются для каждого хоста из группы `access_proxies`. При добавлении нового access-прокси в инвентори -- правило создаётся автоматически.

### Набор правил input

| Приоритет | Правило | Описание |
|---|---|---|
| 1 | `ct state established,related accept` | Разрешение для установленных соединений |
| 2 | `iif "lo" accept` | Разрешение loopback |
| 3 | `tcp dport 22 accept` | SSH доступ |
| 4 | `ip saddr {access_ip} tcp dport {squid_port} accept` | Squid только от access-прокси |
| 5 | `ip saddr {access_ip} tcp dport 443 accept` | MTProxy TLS (условно, при `mtproxy_enabled`) |
| 6 | `icmp type echo-request accept` | Ping IPv4 |
| 7 | `icmpv6 type echo-request accept` | Ping IPv6 |
| default | `policy drop` | Всё остальное отбрасывается |

### Условный блок MTProxy

При `mtproxy_enabled: true` генерируются дополнительные правила, разрешающие доступ от access-прокси на порт 443 (nginx TLS relay → mtg daemon). Правила генерируются динамически аналогично Squid-блоку — для каждого хоста из `access_proxies`.

### Атомарное применение

`flush ruleset` перед загрузкой новых правил обеспечивает атомарную замену: нет промежуточного состояния без правил или с частичными правилами.

## Переменные

| Переменная | Тип | Значение по умолчанию | Описание |
|---|---|---|---|
| `squid_port` | int | -- | Порт Squid для whitelist |
| `groups['access_proxies']` | list | -- | Список access-прокси хостов |
| `hostvars[host]['ansible_host']` | string | -- | IP-адрес access-прокси |
| `mtproxy_enabled` | bool | false | Включение правил для MTProxy port 443 |

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | nftables запущен | `systemctl is-active nftables` | active |
| 2 | Правила загружены | `nft list ruleset` | Таблица inet filter с правилами |
| 3 | SSH доступен | Подключение по SSH | Успешно |
| 4 | Squid доступен от access | curl через Squid с IP access-прокси | 200 OK |
| 5 | Squid недоступен от стороннего IP | curl через Squid с другого IP | Connection refused / timeout |
| 6 | ICMP работает | ping upstream-хост | Ответ |

## Доказательство реализации

### Конструктивное

Шаблон реализует стандартный паттерн hardened firewall: default deny с whitelist. Правила для Squid генерируются динамически из инвентори, обеспечивая синхронизацию firewall с топологией. Атомарное применение через `flush ruleset` исключает race conditions.

### Трассировочное

| C3 | C4 |
|---|---|
| [C3-SU-001](../C3/C3-SU-001.md) | C4-TM-004 -- firewall для upstream-узлов |

### Эмпирическое

- Проверка загрузки: `nft list ruleset` после деплоя
- Проверка доступа: curl через Squid от разных источников
- Проверка блокировки: попытка подключения на Squid-порт от стороннего IP

### Негативное

- **Несанкционированный доступ к Squid** -- policy drop + whitelist по IP
- **Полный lockout** -- SSH (порт 22) всегда разрешён, предотвращая потерю доступа
- **Потеря правил** -- `flush ruleset` + полная перезагрузка конфигурации атомарно
- **IPv6-атаки** -- таблица `inet` (IPv4+IPv6) с единой политикой
