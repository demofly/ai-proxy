# C4-TM-003 -- Шаблон Keepalived (keepalived.conf.j2)

## Ссылки

- Родительские требования C3: [C3-KA-001](../C3/C3-KA-001.md)
- Связанные C4: [C4-PB-001](C4-PB-001.md) (деплоит шаблон), [C4-PB-002](C4-PB-002.md) (источник переменных)

## Описание

Файл `Servers/deploy/templates/keepalived.conf.j2` (~47 строк) -- Jinja2-шаблон конфигурации Keepalived для VRRP-отказоустойчивости access-прокси. Генерирует `/etc/keepalived/keepalived.conf`. Обеспечивает автоматическое переключение виртуального IP-адреса (VIP) между MASTER и BACKUP узлами при отказе Squid.

## Структура файла

| Секция | Строки (прибл.) | Описание |
|---|---|---|
| **global_defs** | 5--9 | router_id, script_user root, enable_script_security |
| **vrrp_script check_squid** | 12--18 | Проверка доступности Squid: `killall -0 squid`, interval=2s, weight=-20, fall=3, rise=2 |
| **vrrp_instance VI_SQUID** | 20--47 | Экземпляр VRRP: state, interface, VRID, priority, authentication, virtual_ipaddress, track_script, notify (условный) |

## Ключевые механизмы

### VRRP Health Check (check_squid)

Скрипт проверки:

```
script "/usr/bin/killall -0 squid"
interval 2
weight -20
fall 3
rise 2
```

- **killall -0 squid** -- проверяет наличие процесса Squid без отправки сигнала
- **interval 2** -- проверка каждые 2 секунды
- **weight -20** -- при неудаче снижает приоритет на 20 (MASTER=100 становится 80, меньше BACKUP=90)
- **fall 3** -- считать мёртвым после 3 неудач (6 секунд)
- **rise 2** -- считать живым после 2 успехов (4 секунды)

### MASTER/BACKUP переключение

- MASTER (priority=100): обслуживает VIP
- BACKUP (priority=90): перехватывает VIP, когда MASTER теряет 20 очков из-за check_squid
- Условие переключения: MASTER effective priority (100-20=80) < BACKUP priority (90)
- Время обнаружения отказа: ~6 секунд (3 * interval=2)

### Настраиваемый интерфейс

```
interface {{ keepalived_interface | default(ansible_default_ipv4.interface) }}
```

Автоопределение через Ansible fact с возможностью ручного переопределения.

### Notify-скрипты (условные)

При определении `keepalived_notify_script` -- вызов внешнего скрипта при смене состояния (master, backup, fault).

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | Конфигурация валидна | Keepalived стартует без ошибок | `systemctl status keepalived` -- active |
| 2 | VIP на MASTER | `ip addr show` на MASTER | VIP присутствует |
| 3 | Failover при остановке Squid | `systemctl stop squid` на MASTER, проверка VIP на BACKUP | VIP мигрирует за ~6 секунд |
| 4 | Failback при запуске Squid | `systemctl start squid` на MASTER | VIP возвращается (preempt) |
| 5 | VRRP-аутентификация | Проверка auth_type PASS в конфигурации | Пароль установлен |

## Доказательство реализации

### Конструктивное

Шаблон реализует VRRP-механизм с health check на основе процесса Squid. Схема weight-based failover: при падении Squid на MASTER его приоритет снижается на 20 (100->80), что ниже BACKUP (90), и VIP мигрирует. При восстановлении -- обратная миграция (preemption).

### Трассировочное

| C3 | C4 |
|---|---|
| [C3-KA-001](../C3/C3-KA-001.md) | C4-TM-003 -- конфигурация VRRP Keepalived |

### Эмпирическое

- Проверка failover: остановка Squid на MASTER, мониторинг миграции VIP
- Проверка failback: запуск Squid на MASTER, мониторинг возврата VIP
- Проверка изоляции: два экземпляра с одинаковым VRID не конфликтуют

### Негативное

- **Отказ Squid** -- check_squid с weight=-20 вызывает failover
- **Split-brain** -- VRRP authentication (PASS) предотвращает подмену
- **Ложные срабатывания** -- fall=3 (6 секунд) предотвращает failover из-за кратковременного сбоя
