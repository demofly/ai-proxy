# C4-RL-002 -- Роль nginx (roles/nginx/)

## Ссылки

## Описание

Директория `Servers/deploy/roles/nginx/` -- Ansible-роль для установки и настройки nginx как SNI-маршрутизатора на порту 443. Обеспечивает co-deployment нескольких сервисов на одном порту: Squid (HTTPS proxy) + MTProxy (Telegram) + другие проекты. nginx читает TLS ClientHello (SNI) без терминации TLS и направляет TCP-соединение на соответствующий backend.

## Ключевые механизмы

### Co-deployment safe архитектура

Модульная архитектура nginx stream позволяет нескольким проектам сосуществовать на порту 443:

```
/etc/nginx/
├── nginx.conf                          # stream { include stream.d/*.conf; }
└── stream.d/
    ├── 00-sni-router.conf              # Shared: SNI map + server listen 443
    ├── 10-aiproxy-upstreams.conf       # Project: MTProxy upstream
    └── sni-map.d/
        └── aiproxy.conf                # Project: SNI -> aiproxy_mtproxy
```

- `00-sni-router.conf` создаётся с `force: no` -- первый deployer создаёт, остальные не перезаписывают
- Каждый проект деплоит свои upstream-определения и SNI map entries
- Fallback: `default fallback_squid` (127.0.0.1:3128) для неизвестных SNI

### SNI routing (00-sni-router.conf)

```
map $ssl_preread_server_name $sni_backend {
    include /etc/nginx/stream.d/sni-map.d/*.conf;
    default fallback_squid;
}

server {
    listen 443;
    ssl_preread on;
    proxy_pass $sni_backend;
    proxy_timeout 86400s;  # 24h для Telegram sessions
}
```

nginx читает SNI из TLS ClientHello (`ssl_preread`) и маршрутизирует TCP-поток на backend. TLS **не** терминируется -- end-to-end шифрование сохраняется.

### SNI map entries (aiproxy.conf)

```
{{ mtproxy_fake_tls_domain }}  aiproxy_mtproxy;
{{ mtproxy_hostname }}  aiproxy_mtproxy;
```

- `example.gov.ru` -> aiproxy_mtproxy (Fake-TLS маскировка)
- `tgproxy.example.com` -> aiproxy_mtproxy (публичный hostname)
- Все остальные -> fallback_squid (Squid на localhost:3128)

### Squid reconfiguration with rollback

Механизм block/rescue для безопасной миграции Squid с порта 443 на localhost:

1. **Backup** squid.conf
2. **Deploy** новый конфиг (behind_nginx, слушает localhost:3128)
3. **Test** nginx -t
4. **Check** если squid на 443 -- restart squid (освободить порт)
5. **Start** nginx
6. **Verify** nginx слушает 443 (wait_for, timeout=10s)

При ошибке на любом шаге -- **rescue**:

1. Восстановить squid.conf из backup
2. Restart squid
3. Fail с сообщением

### Connection limits

```
limit_conn_zone $remote_addr zone=stream_conn_limit:10m;
limit_conn stream_conn_limit 100;
```

Ограничение: 100 одновременных соединений с одного IP.

## Доказательство реализации

### Конструктивное

Роль реализует модульную co-deployment архитектуру nginx stream. SNI routing позволяет мультиплексировать несколько TLS-сервисов на одном порту 443 без терминации TLS. Механизм block/rescue обеспечивает безопасную миграцию Squid с автоматическим откатом при ошибке.

### Трассировочное

### Эмпирическое

- Проверка SNI: `openssl s_client -connect {ip}:443 -servername {domain}`
- Проверка fallback: HTTPS-запрос с неизвестным SNI -> Squid
- Проверка конфигурации: `nginx -t`
- Проверка rollback: намеренная ошибка в конфигурации

### Негативное

- **Потеря доступа при миграции** -- block/rescue с автоматическим rollback Squid
- **Конфликт co-deployment** -- `force: no` для shared SNI router, модульные конфиги
- **DDoS** -- limit_conn 100 per IP
- **Устаревший конфиг** -- миграция: удаление старого monolithic конфига
