# C2-FR-005 — Журналирование и аудит

## Ссылки

- Родительские требования C1: [C1-BC-002](../C1/C1-BC-002.md), [C1-BC-004](../C1/C1-BC-004.md)
- Дочерние требования C3: [C3-SA-001](../C3/C3-SA-001.md), [C3-CS-001](../C3/C3-CS-001.md)

## Описание

Система ведёт структурированный журнал всех проксируемых запросов на уровне **access-прокси**. Журнал используется для аудита, отладки и как источник данных для системы обнаружения угроз CrowdSec. На **upstream-уровне** журналирование **отключено** для обеспечения анонимности ([C2-NF-002](C2-NF-002.md)).

### Формат журнала

Squid access.log на access-прокси использует кастомный `logformat`:

| Поле | Описание | Пример |
|---|---|---|
| `timestamp` | Время запроса (ISO 8601) | 2026-02-23T14:30:00+03:00 |
| `client_ip` | IP-адрес клиента | 192.168.1.100 |
| `request_method` | HTTP-метод | CONNECT |
| `request_url` | Целевой URL/домен | api.anthropic.com:443 |
| `http_status` | Код ответа | 200 |
| `response_size` | Размер ответа (байт) | 4096 |
| `username` | Имя аутентифицированного пользователя | engineer1 |
| `peer_host` | Upstream-прокси, обслуживший запрос | upstream-1.example.com |

### Интеграция с CrowdSec

CrowdSec ([C2-NF-002](C2-NF-002.md)) анализирует access.log в реальном времени для обнаружения:
- Brute-force попыток аутентификации (множественные HTTP 407)
- Аномальных паттернов доступа (сканирование доменов)
- Превышения частоты запросов

При обнаружении угрозы CrowdSec создаёт решение о блокировке, применяемое через nftables bouncer.

### Условное отключение

На upstream-уровне журналирование отключается директивой `access_log none` в конфигурации Squid. Это обеспечивается Ansible-переменной `enable_logging` (по умолчанию `true` на access, `false` на upstream).

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | access.log создаётся на access-прокси | ls -la /var/log/squid/access.log | Файл существует, непустой |
| 2 | Формат записи содержит все обязательные поля | Разбор строки access.log | Все 8 полей присутствуют |
| 3 | Имя пользователя записывается в журнал | Запрос от аутентифицированного пользователя | username != "-" |
| 4 | Журнал отключён на upstream | grep access_log на upstream-ноде | access_log none |
| 5 | CrowdSec читает access.log | cscli metrics | Datasource squid-access активен |

## Доказательство реализации

### Конструктивное

Реализовано в `squid.conf.j2`:
- Используется стандартный Squid native logformat `squid` с полями timestamp, client_ip, method, url, status, size, username, peer_host.
- `access_log daemon:/var/log/squid/access.log squid` на access-уровне.
- `access_log none` на upstream-уровне (при `enable_logging: false`).
- CrowdSec acquis.yaml: datasource type `file`, path `/var/log/squid/access.log`, labels type `squid`.

### Аналитическое

**Кастомный logformat:** стандартные форматы Squid (squid, common) не включают имя пользователя и peer_host, необходимые для аудита и отладки балансировки.

**access_log none на upstream:** анонимизация — upstream не должен хранить данные о запросах пользователей.

### Негативное

| Риск | Митигация |
|---|---|
| Переполнение диска журналами | logrotate с ротацией по размеру/времени (настройка ОС) |
| Отсутствие журналов при сбое Squid | daemon-режим логирования буферизует записи |
| Утечка данных из журнала | Доступ к access.log ограничен правами FS; upstream не ведёт журнал |
