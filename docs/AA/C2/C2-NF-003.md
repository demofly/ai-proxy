# C2-NF-003 — Производительность (SMP, memory tuning)

## Ссылки

- Родительские требования C1: [C1-BC-004](../C1/C1-BC-004.md)
- Дочерние требования C3: [C3-SA-001](../C3/C3-SA-001.md), [C3-SU-001](../C3/C3-SU-001.md)

## Описание

Система оптимизирует производительность проксирования через настройку SMP (Symmetric Multi-Processing) Squid, динамическое управление памятью, увеличение лимитов файловых дескрипторов и тюнинг сетевых параметров ядра. Оптимизация особенно важна для upstream-уровня, обслуживающего долгоживущие соединения к AI API.

### 1. SMP Workers (Squid)

Squid поддерживает многопоточную обработку через SMP workers:

| Параметр | Access-уровень | Upstream-уровень |
|---|---|---|
| `workers` | 1 (по умолчанию) | `ansible_processor_cores` (число ядер CPU) |

На upstream-уровне количество workers соответствует числу CPU-ядер для максимальной утилизации. На access-уровне один worker достаточен при текущей нагрузке (малое количество клиентов).

### 2. Динамическое выделение памяти

| Параметр | Формула | Описание |
|---|---|---|
| `cache_mem` | `ansible_memtotal_mb * 0.2` MB | 20% RAM для in-memory кэша |
| `cache_dir` size | `ansible_memtotal_mb * 0.1` MB | 10% RAM для дискового кэша |
| `maximum_object_size_in_memory` | 512 KB | Максимальный размер объекта в RAM |

Формулы адаптируют конфигурацию к ресурсам конкретного сервера без ручной настройки.

### 3. Лимиты файловых дескрипторов

| Параметр | Значение | Описание |
|---|---|---|
| `max_filedescriptors` | `4096 * workers` (min 65535) | Лимит fd для процесса Squid |
| systemd LimitNOFILE | Соответствует max_filedescriptors | Лимит на уровне systemd unit |
| `/etc/security/limits.conf` | `squid soft/hard nofile` | Лимит на уровне пользователя |

### 4. Таймауты для AI API

| Параметр | Значение | Обоснование |
|---|---|---|
| `read_timeout` | 30 minutes | AI API (Claude, GPT) могут генерировать ответ до 30 минут |
| `connect_timeout` | 60 seconds | Стандартный таймаут подключения |
| `request_timeout` | 0 (disabled) | Без ограничения длительности запроса |
| `persistent_request_timeout` | 2 minutes | Таймаут keep-alive между запросами |

### 5. Sysctl tuning

| Параметр | Значение | Описание |
|---|---|---|
| `net.core.somaxconn` | 65535 | Максимальная очередь подключений |
| `net.netfilter.nf_conntrack_max` | 262144 | Таблица conntrack |
| `net.ipv4.tcp_rmem` | 4096 87380 16777216 | Буферы чтения TCP |
| `net.ipv4.tcp_wmem` | 4096 65536 16777216 | Буферы записи TCP |
| `net.ipv4.ip_local_port_range` | 1024 65535 | Диапазон эфемерных портов |

### Количественные целевые значения

| Метрика | Целевое значение | Метод измерения |
|---|---|---|
| Утилизация CPU (upstream) | < 80% на ядро | top / vmstat |
| Утилизация RAM | < 90% | free -m |
| Время установления CONNECT-туннеля | < 2 секунды | Замер curl --proxy (без TLS handshake с target) |
| Concurrent connections (upstream) | ≥ 1000 на worker | squidclient mgr:info |
| File descriptors in use | < 80% от max_filedescriptors | squidclient mgr:info |

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | SMP workers на upstream = число ядер | grep workers /etc/squid/squid.conf | Значение = ansible_processor_cores |
| 2 | cache_mem соответствует формуле | grep cache_mem /etc/squid/squid.conf | 20% от RAM |
| 3 | max_filedescriptors ≥ 65535 | grep max_filedescriptors squid.conf | ≥ 65535 |
| 4 | read_timeout = 30 минут | grep read_timeout squid.conf | 30 minutes |
| 5 | sysctl значения применены | sysctl net.core.somaxconn | 65535 |

## Доказательство реализации

### Конструктивное

Реализовано в `squid.conf.j2` и Ansible tasks:
- `workers {{ ansible_processor_cores }}` на upstream-уровне.
- `cache_mem {{ (ansible_memtotal_mb * 0.2) | int }} MB`.
- `max_filedescriptors {{ [4096 * workers, 65535] | max }}`.
- Sysctl: Ansible-задача `sysctl:` с `reload: yes`.
- Systemd override: `LimitNOFILE={{ max_filedescriptors }}`.

### Аналитическое

**SMP только на upstream:** access-прокси обслуживает малое количество клиентов (инженерная команда), один worker достаточен. Upstream обслуживает CONNECT-туннели с длительным TLS — SMP критичен.

**read_timeout 30 min:** AI API (особенно Claude с длинными промптами) могут генерировать ответ десятки минут. Стандартный read_timeout (15 min) недостаточен.

**Динамические формулы:** серверы могут иметь разные конфигурации; формулы автоматически адаптируют параметры.

### Негативное

| Риск | Митигация |
|---|---|
| OOM при большом количестве соединений | cache_mem ограничен 20% RAM; ОС OOM killer |
| Исчерпание fd | max_filedescriptors ≥ 65535; мониторинг через squidclient |
| Conntrack table overflow | nf_conntrack_max = 262144; мониторинг |
