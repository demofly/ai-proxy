# C2-FR-003 — Фильтрация доменов (whitelist)

## Ссылки

- Родительские требования C1: [C1-BC-003](../C1/C1-BC-003.md)
- Дочерние требования C3: [C3-SA-001](../C3/C3-SA-001.md)

## Описание

Система реализует доменную фильтрацию по принципу **белого списка** (whitelist): разрешён доступ только к явно указанным доменам. Все остальные CONNECT-запросы отклоняются с HTTP 403 Forbidden.

### Механизм фильтрации

Фильтрация выполняется на **access-прокси** через систему ACL Squid. Два типа правил применяются совместно:

1. **Точное совпадение домена** (`dstdomain`) — для доменов с полным совпадением:
   - Ansible-переменная: `allowed_domains` (список строк)
   - Squid ACL: `acl allowed_domains dstdomain .api.anthropic.com .api.openai.com ...`
   - Точка в начале означает совпадение домена и всех поддоменов

2. **Регулярные выражения** (`url_regex`) — для паттернов:
   - Ansible-переменная: `allowed_domain_patterns` (список regex)
   - Squid ACL: `acl allowed_domain_patterns url_regex -i <pattern>`
   - Применяется для доменов с динамическими поддоменами (например, `.*\.googleapis\.com`)

### Порядок проверки ACL

```
http_access allow authenticated allowed_domains
http_access allow authenticated allowed_domain_patterns
http_access deny all
```

Запрос разрешается, если:
- Пользователь аутентифицирован ([C2-FR-002](C2-FR-002.md)) **И**
- Целевой домен совпадает с `allowed_domains` **ИЛИ** `allowed_domain_patterns`

### Типичный whitelist

| Сервис | Домены / Паттерны |
|---|---|
| Claude (Anthropic) | `.api.anthropic.com`, `.anthropic.com` |
| ChatGPT (OpenAI) | `.api.openai.com`, `.openai.com`, `.chatgpt.com` |
| Gemini (Google) | `.generativelanguage.googleapis.com`, `.aistudio.google.com` |
| GitHub Copilot | `.api.github.com`, `.copilot-proxy.githubusercontent.com` |
| Cursor | `.cursor.sh`, `.*\.cursor\.sh` |
| npm/pip (зависимости) | `.npmjs.org`, `.pypi.org`, `.registry.npmjs.org` |

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | Запрос к разрешённому домену проходит | curl --proxy к api.anthropic.com | HTTP 200 (CONNECT success) |
| 2 | Запрос к запрещённому домену отклоняется | curl --proxy к example.com | HTTP 403 |
| 3 | Regex-паттерн работает для поддоменов | curl --proxy к subdomain.googleapis.com | HTTP 200 |
| 4 | Пустой whitelist блокирует всё | Конфигурация с пустым allowed_domains | Все CONNECT отклонены |
| 5 | Изменение whitelist применяется после reload | Добавить домен, squid -k reconfigure | Новый домен доступен |

## Доказательство реализации

### Конструктивное

Реализовано в Jinja2-шаблоне `squid.conf.j2`:
- Цикл `{% for domain in allowed_domains %}` генерирует ACL `dstdomain`.
- Цикл `{% for pattern in allowed_domain_patterns %}` генерирует ACL `url_regex`.
- Ansible-переменные определены в `group_vars/` и `host_vars/`.

### Аналитическое

**Выбор whitelist (а не blacklist):** принцип наименьших привилегий — разрешено только то, что явно указано. Blacklist неэффективен: невозможно предусмотреть все нежелательные домены. Whitelist гарантирует, что прокси используется только для целевых AI-сервисов.

**Два типа ACL:** точные домены покрывают стабильные эндпоинты, regex — динамические поддомены облачных сервисов (Google Cloud, AWS и др.).

### Негативное

| Риск | Митигация |
|---|---|
| Пользователь обходит whitelist через DNS rebinding | Squid резолвит домен самостоятельно; клиент не контролирует IP |
| Чрезмерно широкий regex разрешает лишние домены | Ревью regex-паттернов; тестирование конфигурации |
| Новый AI-сервис требует добавления в whitelist | Ansible-переменная + переразвёртывание (процесс: [C2-FR-008](C2-FR-008.md)) |
