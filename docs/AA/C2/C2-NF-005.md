# C2-NF-005 — Наблюдаемость

## Ссылки

- Родительские требования C1: [C1-BC-004](../C1/C1-BC-004.md)
- Дочерние требования C3: [C3-SA-001](../C3/C3-SA-001.md), [C3-CS-001](../C3/C3-CS-001.md), [C3-KA-001](../C3/C3-KA-001.md)

## Описание

Система обеспечивает наблюдаемость через логирование, метрики компонентов и мониторинг состояний. Наблюдаемость реализована без централизованного мониторинг-стека (Prometheus/Grafana не входят в scope) — через встроенные средства компонентов.

### Источники наблюдаемости

#### 1. Squid access.log (access-уровень)

Подробно описано в [C2-FR-005](C2-FR-005.md). Ключевые аспекты для наблюдаемости:
- Структурированный формат с обязательными полями.
- Содержит информацию о пользователях, доменах, статусах, upstream-пирах.
- Используется для аудита и анализа использования.

#### 2. CrowdSec метрики и алерты

| Источник | Команда | Информация |
|---|---|---|
| Метрики парсинга | `cscli metrics` | Количество обработанных строк, ошибок парсинга |
| Активные решения | `cscli decisions list` | Заблокированные IP, причины, TTL |
| Алерты | `cscli alerts list` | Сработавшие сценарии, severity |
| Состояние bouncer | `cscli bouncers list` | Статус nftables bouncer |

#### 3. Systemd статусы

| Сервис | Команда | Ноды |
|---|---|---|
| Squid | `systemctl status squid` | access, upstream |
| Keepalived | `systemctl status keepalived` | access |
| CrowdSec | `systemctl status crowdsec` | access, upstream |
| CrowdSec bouncer | `systemctl status crowdsec-firewall-bouncer-nftables` | access, upstream |
| nginx | `systemctl status nginx` | access, upstream |
| mtg | `systemctl status mtg` | upstream |

#### 4. Keepalived VRRP-переходы

Keepalived записывает переходы состояний (MASTER → BACKUP, BACKUP → MASTER) в syslog. Ключевые события:
- `Entering MASTER STATE` — нода стала Master (получила VIP).
- `Entering BACKUP STATE` — нода перешла в Backup (отдала VIP).
- `VRRP_Script(check_squid) failed` — health check Squid не прошёл.

### Количественные целевые значения

| Метрика | Целевое значение | Метод измерения |
|---|---|---|
| Задержка записи в access.log | < 1 секунда | Сравнение timestamp запроса и записи |
| Доступность CrowdSec metrics | 100% (при работающем agent) | cscli metrics exit code |
| Обнаружение отказа Squid через health check | ≤ 15 секунд | interval(5s) × fall(3) |

## Критерии приёмки

| # | Критерий | Метрика / Способ проверки | Целевое значение |
|---|----------|---------------------------|------------------|
| 1 | access.log ведётся на access-нодах | ls -la /var/log/squid/access.log | Файл существует, непустой |
| 2 | cscli metrics возвращает данные | cscli metrics | Ненулевые counters |
| 3 | Все сервисы доступны через systemctl | systemctl status для каждого сервиса | active (running) |
| 4 | VRRP-переходы отражены в syslog | journalctl -u keepalived | Записи о state transitions |
| 5 | Отказ Squid обнаруживается health check | systemctl stop squid; journalctl -u keepalived | VRRP_Script failed |

## Доказательство реализации

### Конструктивное

Реализовано через встроенные механизмы компонентов:
- **Squid:** logformat + access_log в squid.conf.j2.
- **CrowdSec:** cscli CLI для metrics, decisions, alerts.
- **Keepalived:** vrrp_script check_squid; syslog logging.
- **Systemd:** все компоненты запущены как systemd services.

### Аналитическое

**Без централизованного мониторинга:** Prometheus/Grafana/ELK не входят в scope проекта. Текущий масштаб (2+2 ноды) позволяет эффективно мониторить через SSH + встроенные CLI. При росте — рекомендуется добавить Prometheus exporter для Squid и CrowdSec.

**Distributed logging отсутствует:** каждая нода хранит свои логи локально. Для корреляции событий между нодами требуется ручной анализ.

### Негативное

| Риск | Митигация |
|---|---|
| Потеря логов при отказе ноды | Логи на локальном диске; logrotate с retention |
| Нет алертинга при отказе | Keepalived VRRP обеспечивает автоматический failover; ручной мониторинг |
| Рост логов | logrotate; CrowdSec retention policy |
